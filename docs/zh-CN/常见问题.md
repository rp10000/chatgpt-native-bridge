# 常见问题

## 这是 OpenAI API wrapper 吗？

不是。正常流程使用可见的 ChatGPT 网页端和本地桥接器，不需要 API key。

## 它会抓取 ChatGPT 网页吗？

不会。它不读取 ChatGPT 页面、DOM、cookies、localStorage、IndexedDB、token 或浏览器网络请求。

## 现在主路径是什么？

`v1.0.0` 开始，主路径是：

```text
连接 ChatGPT -> 开始处理 -> 查看结果 -> 交给 Codex
```

真正能读取本地项目、运行命令、修改文件并写回 Codex 的路径是 ChatGPT Thinking/MCP。

如果当前 ChatGPT 模式支持 MCP Apps UI，关键工具调用还会在 ChatGPT 网页对话里显示项目、命令、文件变更和 Codex 写回卡片。

## GPT-5.5 Pro 用来做什么？

Pro 只是辅助规划。它不能直接读取本地文件，也不能在这个设计里直接调用 bridge 工具。

客户端会把项目上下文打包复制给 Pro，再把带标记的 Pro 回复导入 Codex inbox。

## 为什么不自动操作浏览器？

浏览器自动化不稳定，也会模糊安全边界。这个项目保留用户对 ChatGPT 网页会话的可见控制。

## 为什么不能自动创建 ChatGPT app？

自动创建或修改 ChatGPT app 需要浏览器自动化，或者调用未公开的网页接口。这个项目不做这些事。

客户端会准备好连接地址和下一步话术，把步骤尽量压短。

## MCP 会给 ChatGPT shell 权限吗？

会。MCP 工作区路径暴露 `bash` 工具，用于当前连接项目。它也暴露 `read`、`write`、`edit`、`list_directory`、`search_workspace`、`command_history`、`show_changes`。

它不会自动 commit 或 push。REST Actions fallback 不暴露 shell 或源码写入工具。

## 还能手动使用吗？

可以：

```bash
cgn handoff --task "复核这个项目"
cgn done
```

## 什么内容不要交给 ChatGPT？

不要发送 `.env`、私钥、cookies、session dump、Authorization header、token、密码，或任何你不愿意交给 ChatGPT 的内容。
